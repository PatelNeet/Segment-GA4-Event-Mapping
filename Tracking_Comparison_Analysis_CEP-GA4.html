2 — CEP events mapped to GA4 but absent from GA4→CEP doc

2 events

CEP→GA4 defines these GA4 event mappings but there is no corresponding GTM tag documented in the GA4→CEP sheet. Either the GTM tag hasn't been created yet, or it was omitted from the GA4→CEP doc.

add_payment_info — mapped by CEP "Checkout Step Completed" (conditional on label containing "payment methods"), but completely missing from the GA4→CEP tag inventory. The add_shipping_info counterpart IS in the GA4→CEP doc — so this is likely a documentation gap rather than an implementation gap.
file_download — CEP maps "File Downloaded" to this GA4 auto-collected event, but no corresponding GTM tag appears in the GA4→CEP doc. Since file_download is auto-collected in GA4 (when enabled), there may intentionally be no GTM tag — but it should at minimum be noted in the GA4→CEP sheet.

Chat Interaction is the biggest GA4 blind spot. 27 CEP events defined, only 3 have GA4 mappings (session started, user message, chat closed). The other 24 — including all feedback loops, escalation, flow steps, CTA views, and privacy consent — are invisible in GA4. This makes it impossible to measure bot funnel conversion or session quality in GA4.
Sign-In & Account: 12 CEP events, only login (User Sign-In Succeeded) lands in GA4. All 11 remaining events — failures, dismissals, account creation, password reset — are Segment-only. You can see the success but not the failure modes in GA4.
Experimentation framework: All 3 events (Experiment Viewed, Flag Viewed, Clicked) have no GA4 mapping. A/B test exposure data is entirely absent from GA4, making GA4-based experiment analysis reliant on external stitching rather than native events.
Survey & Ratings new frameworks: Survey Submission (2 events, 0 GA4) and 3 of 4 Ratings & Reviews events have no GA4 mapping. These are newly defined frameworks still awaiting GA4 implementation.

4 — Naming inconsistencies & conflicts between the two docs

5 cases

The same underlying mapping is described with different CEP event names across the two documents. These are mostly cosmetic (slash vs space differences) but one — sort_by — is a genuine conflict.

Fly-Out naming drift (4 events): GA4→CEP uses "Fly-Out Viewed / Pop-Up Viewed", "Fly-Out Closed / Pop-Up Closed", "Fly-Out Clicked / Pop-Up Clicked" — CEP→GA4 uses "Fly-Out / Pop-Up Viewed", "Fly-Out / Pop-Up Closed", "Fly-Out / Pop-Up Clicked". The word order differs. Same for artwork_tips_popup and paypal_paylater_popup. Low risk but will cause string-match issues in any automated tooling.
contact_us — negative feedback event name differs: GA4→CEP calls it "Feedback Component Engaged (when negative feedback is submitted with a response)" — CEP→GA4 calls it "Feedback Component Engaged (negative)". Same event, very different names.
sort_by — genuine conflict: CEP→GA4 maps "Fly-Out / Pop-Up Clicked" to sort_by as one of its GA4 events. GA4→CEP marks sort_by as "No direct CEP mapping". The two docs directly contradict each other on this mapping. Needs a single source-of-truth decision.

5 — Other interesting observations

pdp_interaction is fired by 5 different GTM tags — Tier Select, Attr_Select, Prod Text, Reviews-Top, and a generic PDP Interaction tag. All five route to the same GA4 event. On the CEP side it maps to both "Attribute Selected" and "Ratings Reviews Engaged". This single event name is doing a lot of work: product options, text editing, review CTA clicks, and tier selection all collapse into one. This is a reporting clarity risk — knowing that pdp_interaction fired tells you very little without gtm_event_sub_name.
Recommendation: ensure gtm_event_sub_name is always populated so this event can be meaningfully segmented in BigQuery / GA4 explorations.
Chat Interaction coverage asymmetry is by design — but worth calling out: GA4→CEP shows chat_interaction mapping to 3 CEP events (session start, user message, chat close). CEP→GA4 shows these same 3 GA4 mappings but defines 24 additional events that are Segment-only. The clear implication is that GA4 is intentionally receiving only a coarse-grained chat signal, with full session-level detail living in Segment/BQ. This is a valid architectural choice but it means GA4 cannot answer "did the bot resolve the issue?" without a data join.
add_shipping_info is in GA4→CEP but add_payment_info is not. Both are fired by the same CEP event (Checkout Step Completed) with a label condition. The shipping step has a GTM tag but the payment step does not appear in the GA4→CEP sheet. Either the tag was missed in documentation or the payment step uses a different mechanism (checkout_payment_interaction, which has no CEP mapping either). The checkout funnel is incomplete in both directions.
Three entirely new frameworks (Chat, Survey, Ratings & Reviews) are CEP-defined but minimally wired to GA4. This pattern is consistent with a "CEP-first, GA4-second" rollout — the event taxonomy is defined before the GA4 implementation is complete. Worth tracking as a migration backlog.
The ecommerce core (Product Funnel) is the most complete and consistent area. view_item_list → view_item → select_item → add_to_cart → begin_checkout → add_shipping_info → purchase all appear in both documents with matching names. The only gap is Checkout Step Viewed (intentionally no GA4 event) and the add_payment_info asymmetry above.
4 consent/privacy events fire in production with no CEP path. Given that onetrust_consent is a global GA4 param, and the CEP framework explicitly includes it in all events, the absence of dedicated CEP events for consent_engagement, do_not_track, and dns_form_interaction is a gap worth flagging to compliance stakeholders.
